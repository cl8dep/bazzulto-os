# Keyboard Technical Debt

## Console UTF-8 buffer is not thread-safe

`console_putc()` accumulates multi-byte UTF-8 sequences in a static buffer
(`utf8_buffer`). If two threads or interrupt handlers call `console_putc()`
concurrently, bytes from different codepoints can interleave and corrupt
the buffer.

**Fix:** Add a spinlock around the UTF-8 accumulation logic in `console_putc()`.
This becomes critical when SMP support is added.

**Affected file:** `kernel/drivers/console/console.c`

---

## Key repeat rate is not configurable from the guest

The virtio-input protocol does not expose key repeat rate/delay to the guest.
Repeat events (`value=2`) are generated by QEMU at a fixed rate controlled by
the host. There is no way for Bazzulto to adjust repeat speed.

**When this matters:** When running on real hardware with a USB HID keyboard,
the USB HID protocol does allow setting repeat rate via `SET_IDLE` request.
The USB HID driver should expose this when implemented.

**Affected file:** `kernel/drivers/keyboard/keyboard.c`

---

## Font glyphs need visual validation

The Latin Extended glyph bitmaps in `font_latin_extended.c` were generated
programmatically, not copied from a verified public domain font. Some characters
may render incorrectly. Each glyph should be verified visually on the actual
framebuffer console.

**Fix:** Replace with glyphs from a verified source (e.g. GNU Unifont, Cozette,
or the Terminus font, all of which are open source).

**Affected file:** `kernel/drivers/console/font_latin_extended.c`

---

## Parser does not report error details

`keymap_parse()` returns `-N` where N is the number of errors, but does not
report which lines failed or why. In a kernel context without `printf` available
at parse time, this is acceptable. When the VFS and logging are available, the
parser should log line numbers and error descriptions via the kernel log.

**Affected file:** `kernel/drivers/keyboard/keymap.c`
